#### OX | [637]
JWT Payload는 암호화되어 있어 민감한 정보를 담아도 안전하다.
> X
> JWT Payload는 Base64 인코딩이므로 누구나 디코딩할 수 있다. 비밀번호, 개인정보 등 민감한 정보를 담으면 안 된다.

#### OX | [638]
분산 환경에서 세션 외부 저장소로 Redis가 RDB보다 적합하다.
> O
> Redis는 인메모리 캐시로 I/O 비용이 낮다. RDB는 디스크 기반이라 세션 조회 비용이 크고 Scale-out 환경에서 병목이 생길 수 있다.

#### 빈칸 | [639]
JS에서 쿠키에 접근하지 못하도록 막는 쿠키 속성은 ___ 이다.
> HttpOnly
> HttpOnly 속성이 설정된 쿠키는 JavaScript의 document.cookie로 접근할 수 없다. XSS 공격으로 쿠키를 탈취하는 것을 방지한다.

#### 빈칸 | [640]
JWT에서 토큰 위변조를 방지하기 위해 헤더+페이로드를 비밀키로 서명한 값을 ___ 이라고 한다.
> Signature
> Signature는 서버만 알고 있는 비밀키로 생성된다. 클라이언트가 토큰을 조작하면 서버가 Signature 검증 시 불일치를 감지할 수 있다.

#### 객관식 | [641]
세션 방식과 JWT 방식의 차이로 올바른 것은?
1. ✅ 세션은 서버에 상태를 저장하고, JWT는 무상태로 서버가 상태를 저장하지 않는다
2. JWT는 서버에 사용자 상태를 저장한다
3. 세션은 탈취 시 방어가 어렵고 JWT는 완벽하게 방어할 수 있다
4. 둘 다 동일한 방식이다
> 세션은 서버 저장소에 상태를 두므로 Scale-out 시 공유 저장소가 필요하다. JWT는 무상태이나 탈취 시 만료 전까지 방어가 어려워 Refresh Token 전략을 보완으로 사용한다.

#### 객관식 | [642]
Access Token과 Refresh Token을 분리하는 주목적은?
1. 서버 부하를 줄이기 위해
2. 토큰 크기를 줄이기 위해
3. ✅ Access Token 유효시간을 짧게 설정해 탈취 피해를 최소화하기 위해
4. 암호화 강도를 높이기 위해
> ATK 유효시간을 짧게(수십 분)하면 탈취되어도 피해 시간이 제한된다. RTK로 ATK를 재발급받는 방식으로 사용자 경험을 유지하면서 보안을 강화한다.
