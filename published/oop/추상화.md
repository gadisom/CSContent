> Swift는 abstract class 없이 Protocol과 Extension 기본 구현으로 추상화한다. some은 opaque(타입 고정), any는 existential(타입 소거)이다.

## 정의
- 추상화: 구현 세부사항을 숨기고 인터페이스(무엇을 할 수 있는가)만 외부에 노출하는 OOP 원칙
- Protocol: Swift의 핵심 추상화 도구. 타입이 반드시 구현해야 할 요구사항(메서드·프로퍼티)을 정의. class·struct·enum 모두 채택 가능
- Protocol Extension: Protocol에 기본 구현을 제공하는 기능. Java abstract class의 기본 메서드와 유사하지만 상속 없이 동작
- Opaque Type(some): 컴파일 타임에 타입이 고정되지만 호출 측에 타입 정보를 노출하지 않는 반환 방식. `some View` 등에 사용
- Existential Type(any): 런타임에 다양한 타입을 담을 수 있는 타입 소거 박스. Swift 5.7부터 `any Protocol` 명시 필수
- associated type: Protocol 내부에서 사용할 타입을 추상화하는 제네릭 플레이스홀더. Equatable의 Self 등이 대표 예시

## 핵심 포인트
- Swift에는 abstract class 키워드가 없다. Protocol + Extension 기본 구현으로 동일한 역할을 수행한다.
- some Protocol은 컴파일 타임에 단일 구체 타입이 확정되므로 성능 오버헤드가 없다. SwiftUI의 body: some View가 대표 사례다.
- any Protocol은 여러 타입을 한 배열에 담아야 할 때 사용하지만, 박싱 비용이 발생하고 associated type이 있는 Protocol엔 사용 불가했다(Swift 5.7 이후 일부 완화).
- Protocol extension으로 기본 구현을 제공하면 채택 타입은 필요한 부분만 오버라이드하면 된다.
- Dependency Injection 시 Protocol 타입으로 의존성을 주입하면 구현체를 갈아끼울 수 있어 테스트 용이성이 높아진다.
