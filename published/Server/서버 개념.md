- 메시지큐
    - 대표적인 RabbitMQ, Kafka
    - RabbitMQ
        - AMQP 프로토콜 사용
            - Excahnge
                - 프로듀서에게 메시지를 수신하는 곳
            - Queue
                - 메모리나 디스크에 메시지 저장
            - Binding
                - Exchange와 메시지큐의 맵핑 관리
        - AMQP를 구현한 MQ들
            - RabbitMQ, ZeroMQ, ActiveMQ
            - RabbitMQ는 경량 오픈소스 메시지 브로커, 구현 스펙이 간단함, 비동기 처리를 쉽게 적용할 수 있다는 장점
        - 메시지를 큐 형태로 저장 → 선입선출, 컨슈머가 메시지를 받아갈 때마다 큐에서 제거하는 일도 담당함
        - 메시지가 주로 메모리에 저장되어 처리가 빠르지만 유실 위험성이 있음
        - 큐로 메시지를 구분지을 수 있음 → 특정 큐나 모든 큐로 메시지를 보낼 수 있어 다양한 메시지를 처리할 때 카프카보다 유리함, 컨슈머가 성공적으로 메시지를 처리했는지 여부를 브로커에게 전달함으로써 신뢰성 있는 전달을 보장할 수 있음 → 이 때문에 트랜잭션과 같은 안정적인 처리가 필요한 작업에 적합
    - Kafka
        - TCP 기반의 자체 프로토콜 사용
        - 메시지를 디스크에 로그 형태로 저장 → 컨슈머가 메시지를 받아간다고 지워지지 않음
        - 배치 기능을 제공하기 때문에 메시지를 여럿씩 받아오고 보내주는 일도 가능하여 RabbitMQ에 비해 빠르고 간편
        - 토픽으로 메시지 그룹을 구분지을 수 있음
        - 로그 안에 메시지는 오프셋을 갖고 있으며, 컨슈머는 마지막 오프셋을 기억하여 중복 없이 다음 메시지를 요청함
        - 컨슈머를 구현하는 일이 RabbitMQ에 비해 까다로울 수 있음
        - 메시지 유실로부터 안전함 → 컨슈머의 요청에 따라 재전송이나 특정 메시지 전송 등도 가능함
        - 브로커를 여러 서버에 분산하여 운영하는 클러스터링에 보다 유리함

- JDBC 이해
    - 데이터베이스마다 사용법이 다름
    - 따라서 JDBC 표준 인터페이스가 등장, 이는 자바에서 데이터에스에 접속할 수 있도록 하는 자바 API
    - 이 인터페이스를 각 데이터베이스 드라이버가 구현, 드라이버가 실제 데이터베이스와 통신
    - 하지만 각 데이터베이스마다 SQL, 데이터 타입 등 일부 사용법이 다름 → SQL Mapper와 ORM의 등장
    - SQL Mapper(JdbcTemplate, MyBatis)
        - 개발자가 SQL을 직접 작성
    - ORM(JPA)
        - 하이버네이트는 JPA의 구현체, 반복적인 SQL 작성 줄임
    - DriverManager 클래스를 제공하여 각 데이터베이스로부터 커넥션을 획득할 수 있도록 기능을 제공함
- 커넥션 풀의 등장
    - 매번 커넥션을 맺고 끊으면서 tcp 연결 수립 필요
    - DriverManager를 사용하다가 HikariCp를 사용하려면 기존 코드를 수정해야 하는 문제 발생
    - 따라서 DataSource라는 인터페이스를 제공하여 커넥션을 획득하는 방법을 추상화함
- 트랜잭션 처리 방법
    - 트랜잭션 처리 방식이 JDBC, JPA, 하이버네이트 등 각각 다름
    - 따라서 트랜잭션 기능을 추상화함
- 트랜잭션 동기화
    - 함수를 호출할 때마다 같은 커넥션을 유지한 채 동기화되어야 함
    - 매개변수로 커넥션 객체를 넘기지 않도록 하고자 **트랜잭션 동기화 매니저** 등장
    - ThreadLocal을 사용해서 커넥션을 동기화
- 트랜잭션 전파
    - REQUIRED 전파 옵션을 사용하여 물리, 논리 트랜잭션을 나눌 수 있음
    - 하나의 논리 트랜잭션이 롤백되면 물리 트랜잭션도 롤백
    - 트랜잭션 동기화 매니저를 통해 rollbackOnly 값을 수정하여 롤백 여부를 마킹함

- 예외 처리 흐름(@ControllerAdvice)
    - 이 어노테이션은 @Component를 포함, 즉 스프링 빈으로 등록됨
    - 디스패처 서블릿이 에러를 catch함
    - 해당 에러를 처리할 수 있는 처리기(HandlerExceptionResolver의 구현체들)가 에러를 처리함
        - ControllerAdvice는 ExceptionHandlerExceptionResolver에 의해 처리됨
    - 컨트롤러의 ExceptionHandler로 처리가능한지 검사함(Controller 메서드에 선언되어 있는지 판단) → 처리가 가능하면 ServletInvocableHandlerMethod를 만들어서 반환, 이때 예외를 처리할 빈으로 Controller를 주입
    - 처리가 불가하다면 등록된 모든 ControllerAdvice 빈을 검사함 → 처리 가능한 ControllerAdvice에 등록된 ExceptionHandler가 있다면 ServletInvocableHandlerMethod를 만들어서 반환, 이때 예외를 처리할 빈으로 ControllerAdvice 빈을 주입
    - 반환받은 ServletInvocableHandlerMethod에는 ExceptionHandler를 갖는 빈과 ExceptionHandler의 구현 메소드가 존재 → 리플렉션으로 ExceptionHandler의 구현 메소드를 호출하여 예외를 처리함
    - **CGLIB, JDK 동적 프록시를 사용하지 않기 때문에 AOP 프록시를 만드는 것은 아니지만 방식이 비슷함**
- 스프링 빈 등록 과정

    - **빈 저장소에 등록하기 전 빈 후처리기에게 먼저 전달**
    - 전달된 빈을 조작하거나 다른 객체로 바꿔칠 수 있음
    - 빈 후처리기가 빈을 반환하고 이를 빈 저장소에 등록

    - 모든 어드바이저(포인트컷 + 어드바이스) `@Aspect로 생성 할 수 있음` 를 조회하고 포인트컷을 참고하여 현재 빈이 프록시 적용 대상인지 판단하면서 프록시 객체를 생성

- Filter vs Interceptor
    - Filter
        - 서블릿 컨테이너 레벨에서 동작
        - Spring MVC의 앞단
        - 핸들러를 알 수 없어 URL마다 권한 정보를 알 수 없음
        - 인증/보안 관련 처리
        
        ```java
        public class MyFilter implements Filter {
            public void doFilter(...) { ... }
        }
        ```
        
    - Interceptor
        - Spring MVC 레벨에서 동작
        - 핸들러를 알 수 있어서 Controller마다 접근 권한을 확인할 수 있음
        - preHandle: 컨트롤러 실행 직전 호출
        - PostHandle: 컨트롤러 실행 후, View 렌더링 전, 컨트롤러 메서드가 정상적으로 리턴한 경우
        - afterCompletion: 예외 발생 여부와 상관없이 무조건 호출됨, View 렌더링은 끝난 뒤라 응답 조작은 불가
        - 인증/인가 로직(특히 인가)
        - 로그인 체크
        - 공통 로깅
        - 모델 조작 가능
        
        ```java
        public class MyInterceptor implements HandlerInterceptor {
            public boolean preHandle(...)
            public void postHandle(...)
            public void afterCompletion(...)
        }
        ```
        
- Spring MVC 요청이 처리되는 과정
    - Dispatcher Servlet이 요청을 받음
    - URL을 기준으로 요청을 처리할 수 있는 핸들러(Controller) 조회
    - 찾은 핸들러(Controller)를 실행할 수 있는 핸들러 어댑터 조회
        - 핸들러 어댑터에게 핸들러 호출을 위임하는 이유 → 유연성과 확장성을 확보하기 위함
        - 스프링에는 다양한 타입의 핸들러가 존재해왔음(@RequestMapping 기반, Controller 인터페이스 기반, HttpRequestHandler..), 이 모든 타입을 판별한다면 코드 복잡성이 올라감
        - 따라서 각각의 핸들러를 처리할 수 있는 어댑터를 따로 두고 어댑터를 호출하는 방식으로 유연성을 확보함
        - 핸들러 어댑터는 핸들러 타입별 호출 방법을 담당
            - RequestMappingHandlerAdapter: @RequestMapping, @GetMapping 메소드 호출 처리, 파라미터 바인딩, @RequestBody, @ResponseBody, Validation 등 담당
            - HttpRequestHandlerAdapter: HttpRequestHandler를 호출하는 기능 담당
            - SimpleServletHandlerAdapter: Servlet을 직접 호출할 때 사용하는 어댑터
    - 핸들러 어댑터가 핸들러를 호출하기 전, Controller 메서드에 선언된 매개변수 정보를 리플렉션으로 조회하고 각 어노테이션마다 적절한 ArgumentResolver를 사용하여 HTTP 메시지를 자바 객체, 정수 등으로 변환하여 수집한 뒤, 핸들러 어댑터가 핸들러를 호출
        - ArgumentResolver가 @RequestBody, HttpEntity, @PathVariable 등 어노테이션 정보를 리플렉션으로 보고 적절한 ArgumentResolver 구현체를 호출
        - ArgumentResolver 구현체 내부에서 HttpMessageConverter를 사용하여 적절한 값을 생성하고 반환
        - 매개변수를 모두 모아서 핸들러 어댑터는 핸들러 호출(Controller)
        - 응답이 반환되면 타입을 확인하고 등록된 ReturnValueHandler 목록 탐색
        - 적절한 MethodProcessor 구현체를 찾아 HttpMessageConverter로 변환하는 작업 위임
    - 이제 응답을 Dispathcer Servlet으로 반환하고 끝

- 쿠키 vs 세션 vs 토큰
    - 쿠키
        - 로컬에 저장되는 키와 값이 들어있는 작은 데이터 파일
        - 최대 300개 쿠키 저장 가능 & 하나의 도메인 당 20개의 값만 가질 수 있음
        - Response Header에 Set-Cookie 속성을 사용하여 클라이언트에 쿠키를 만들 수 있음
        - 탈취 가능, 조작 가능 → HttpOnly 와 Secure를 사용할 것, HttpOnly는 JS로 쿠키 접근 불가, Secure를 사용하면 HTTPS 통신에서만 전달
        - 세션 쿠키(브라우저 종료 시 삭제), 지속 쿠키(만료 시간동안 저장됨)
    - 세션
        - 쿠키 기반이지만 서버 측에서 관리
        - 클라이언트를 구분하기 위해 세션 ID를 부여하며 클라이언트가 브라우저를 종료할 때까지 인증상태를 유지
        - 서버에서 임의로 세션 폐기 가능 → 보안 우수
        - 서버 상태 관리 필요 → Scale-out 비용 증가
    - 토큰
        - 무상태성
        - 서버는 사용자의 상태를 관리하지 않고 토큰의 유효성만 판단하면 됨
        - 토큰 탈취하면 방어 어려움
        - 이를 보완하고자 Refresh Token을 도입해서 완전한 무상태성은 포기를 하고 보안을 높임, ATK 유효시간을 짧게 설정하고 RTK로 재발급 요청을 받도록 하여 관리
        - 하지만 RTK 탈취 가능성이 있긴해서 완벽한 보안을 보장받을 수 없음
- JWT 토큰
    - 헤더: JWT가 어떤 방식으로 서명되었는지 정보를 담는다. Base64 인코딩
    - 페이로드: 유저 정보와 토큰의 claim을 담는 부분. Base64 인코딩. 민감한 정보를 담으면 안됨
    - 서명: 헤더 + 페이로드를 비밀키로 서명한 값. 토큰 위변조 방지. 서버만 비밀키를 알고 있어서 조작 여부를 서버가 확인할 수 있음
- 분산 환경에서의 세션 관리
    - 서버가 세션 저장소에 데이터를 저장하고 조회하므로 스케일 아웃되면 문제가 발생함
    - 서버A에 생성된 세션을 서버B가 모르고 로그인이 끊길 수 있음
    - 따라서 외부 저장소로 세션을 빼야 하는데 RDB를 사용하면 I/O 비용이 크므로 적절하지 않음
    - Redis같은 인메모리 캐시를 사용할 수 있음
- AOP
    - JDK 동적 프록시
        - 인터페이스 기반 프록시 생성
        - InvocationHandler 구현체로 부가 로직 추가한 프록시 객체 생성
    - CGLIB
        - 클래스 기반 프록시 생성
        - MethodInterceptor 구현체로 부가 로직 추가한 프록시 객체 생성
        - 그런데 클래스 기반이기 때문에 원본 객체를 생성하고, 프록시 객체를 생성할 때 부모 클래스의 기본 생성자를 반드시 호출해야 하는 문제가 있었음(생성자 2번 호출 문제) → 스프링 4.0부터 `objenesis` 라는 라이브러리가 추가되어 기본 생성자 없이도 객체 생성이 가능하다록 해결됨
    - Spring AOP
        - 두 방식을 신경쓰지 않고도 부가 기능을 사용하도록 Advice 개념을 도임
        - ProxyFactory 클래스가 프록시 생성을 담당, 내부에서 Advice를 호출하는 전용 InvocationHandler, MethodInterceptor를 구현하여 갖고 있음
        - 만약 CGLIB 고정으로 사용하려면 `proxyFactory.setProxyTargetClass(true)` 지정해야 함, 구체 클래스를 상속받는 프록시 객체를 만들어 줌
        - Spring AOP는 항상 CGLIB을 사용해서 구체클래스 기반으로 프록시를 생성함
        - 개발자는 Advice만 구현하면 됨
- 포인트컷, 어드바이스, 어드바이저
    - 포인트컷
        - 어디에 부가 기능을 적용할지 말지 정하는 필터링 로직
    - 어드바이스
        - 프록시가 호출하는 부가 기능, 프록시 로직이라고 생각
    - 어드바이저
        - 하나의 포인트컷과 하나의 어드바이스를 갖고 있는 것
- Spring이 제공하는 포인트컷
    - NameMatchMethodPointcut, JdkRegexMethodPointcut, TruePointcut, AnnotationMatchingPointcut, AspectJExpressionPointcut 등 많은 포인트컷을 제공함
    - **제일 중요한 것은 aspectJ 표현식, 실무에서 가장 많이 사용하는 방식**
- 여러 어드바이저 적용하기
    - 어드바이저 여러 개를 붙이려고 한다. 그렇다면 ProxyFactory를 두 번 호출해서 어드바이저를 적용한 프록시 객체를 체이닝 하면 됨
    - 그런데 어드바이저가 10개라면 10개의 프록시를 만들어야 함 → 비효율
    - 프록시 팩토리는 여러 어드바이저를 적용할 수 있도록 기능을 제공함
    - proxyFactory.addAdvisor(..)를 여러 번 호출하여 복수 개의 어드바이저를 등록할 수 있음
    - **즉 하나의 target에 여러 AOP가 적영되어도 하나의 프록시만 생성함**
- ProxyFactory를 사용할 때의 문제점
    - 프록시에 포인트컷과 어드바이저를 쉽게 추가할 수 있게 되었음
    - 하지만 설정 파일이 지나치게 많음 → AOP를 적용하려면 수동으로 빈 등록을 해야 함
    - 컴포넌트 스캔을 상요하는 클래스에는 자동으로 빈 등록이 되므로 지금까지의 방법으로 프록시를 만들 수 없음
    - **이 문제를 빈 후처리기가 해결**
- 빈 후처리기
    - 컴포넌트 스캔, @Bean을 스프링은 대상 객체를 생성하고 컨테이너 내부 빈 저장소에 등록
